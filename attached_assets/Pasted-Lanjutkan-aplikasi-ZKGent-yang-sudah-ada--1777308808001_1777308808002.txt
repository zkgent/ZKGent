Lanjutkan aplikasi ZKGent yang sudah ada.                                     
                                                                                
  Fokus tahap ini HARUS sempit dan disiplin.                                    
  Jangan melebar ke redesign, refactor kosmetik, atau fitur sampingan.          
                                                                                
  TUJUAN UTAMA TAHAP INI:                                                       
  Menutup 3 celah kritis yang masih tersisa:                                    
  1. dependency integrity harus benar                                           
  2. wallet flow harus naik dari signMessage ke signTransaction nyata           
  3. istilah dan arsitektur proof harus jujur dan user-centric, bukan           
admin-centric                                                                   
                                                                                
  TAMBAHAN PENTING:                                                             
  - sistem harus bergerak ke model USER-BY-USER, bukan admin-only view          
  - setiap user harus punya identitasnya sendiri setelah connect wallet         
  - arsitektur harus mengarah ke penggunaan MAINNET sebagai target utama        
  - boleh tetap menyediakan fallback/test mode, tapi default design dan copy    
harus mengarah ke mainnet readiness                                             
  - jangan merusak landing page                                                 
  - jangan redesign marketing page                                              
  - jangan mengacak modul di luar scope ini

  ATURAN KERJA                                                                  
  ================================                                              
                                                                                
  - High agency                                                                 
  - Kerjakan langsung                                                           
  - Jangan kabur ke area lain                                                   
  - Jangan overclaim                                                            
  - Jangan menyebut sesuatu “ZK proof” jika itu bukan zero-knowledge proof      
  - Jangan menyebut sesuatu “wallet signing implemented” jika masih

  ================================                                              
  PRIORITAS 1 — DEPENDENCY INTEGRITY                                            
  ================================                                              
                                                                                
  Rapikan dependency dan build integrity sekarang juga.                         
                                                                                
  Kerjakan:                                                                     
  1. periksa semua import yang saat ini digunakan, terutama:                    
     - @solana/web3.js                                                          
     - @noble/curves                                                            
     - library wallet yang benar-benar dipakai                                  
  2. pastikan semua dependency itu benar-benar ada di:                          
     - package.json                                                             
     - lockfile                                                                 
  3. hapus mismatch antara code dan dependency state                            
  4. pastikan fresh install bisa build                                          
  5. jika perlu, tambahkan script verifikasi sederhana                          
  6. jangan tinggalkan code yang mengandalkan package yang tidak tercatat       
                                                                                
  Output yang diharapkan:                                                       
  - dependency bersih                                                           
  - import valid                                                                
  - build reproducible                                                          
  - tidak ada “works only in current environment” trap                          
                                                                                
  ================================                                              
  PRIORITAS 2 — USER-BY-USER IDENTITY MODEL                                     
  ================================                                              
                                                                                
  ZKGent harus bergerak dari operator/admin-centric menuju                      
wallet-identity-centric.                                                        
                                                                                
  Kerjakan:                                                                     
  1. buat model identitas user berbasis wallet                                  
  2. setiap wallet yang connect harus direpresentasikan sebagai user identity   
yang unik                                                                       
  3. simpan identity minimal:                                                   
     - wallet address                                                           
     - fingerprint / derived identity id                                        
     - first_seen_at                                                            
     - last_seen_at                                                             
     - network preference                                                       
     - signing status / connection metadata                                     
  4. relasikan note / settlement / signing request / activity ke identity user  
yang relevan                                                                    
  5. ubah asumsi “operator melakukan semua” menjadi model:                      
     - operator/system role tetap ada bila perlu                                
     - tetapi transaksi dan signing harus bisa ditelusuri per-user              
  6. dashboard harus menampilkan state user-aware, bukan hanya global/admin view
  7. jika perlu, tambahkan API:                                                 
     - current wallet identity                                                  
     - wallet session status                                                    
     - user-linked settlements                                                  
     - user-linked notes / transactions                                         
  8. jangan buat auth web2 aneh-aneh, gunakan wallet identity sebagai pusat     
                                                                                
  Yang penting:                                                                 
  - setiap user punya identitas sendiri setelah connect wallet                  
  - jangan lagi berpikir sistem ini murni admin panel                           
                                                                                
  ================================                                              
  PRIORITAS 3 — SIGNTRANSACTION NYATA                                           
  ================================                                              
                                                                                
  Wallet flow harus naik level dari signMessage menjadi signTransaction yang    
benar.                                                                          
                                                                                
  Kerjakan:                                                                     
  1. implement browser wallet transaction signing yang nyata                    
  2. alurnya harus menjadi:                                                     
     - backend/build layer menyiapkan transaction nyata                         
     - frontend wallet connect                                                  
     - wallet melakukan signTransaction atau signAndSendTransaction             
     - backend menerima hasil yang relevan                                      
     - settlement lifecycle diupdate berdasarkan hasil nyata                    
  3. jangan pakai base64 JSON pura-pura sebagai “tx_data”                       
  4. jika wallet provider mengharuskan frontend submit langsung ke chain, desain
flow itu dengan benar                                                           
  5. jika backend yang submit, pastikan serialized transaction/signature flow   
valid                                                                           
  6. status settlement harus mencerminkan:                                      
     - signing_requested                                                        
     - transaction_prepared                                                     
     - signed                                                                   
     - submitted_on_chain                                                       
     - confirmed                                                                
     - finalized                                                                
     - failed                                                                   
  7. dashboard harus menampilkan:                                               
     - wallet connected/disconnected                                            
     - current user wallet                                                      
     - pending signing requests user                                            
     - latest signed transactions                                               
     - per-user on-chain status                                                 
                                                                                
  Catatan:                                                                      
  - signMessage boleh tetap ada untuk testing/debugging                         
  - tapi jangan jadikan itu jalur utama settlement                              
                                                                                
  ================================                                              
  PRIORITAS 4 — MAINNET-FIRST ARCHITECTURE                                      
  ================================                                              
                                                                                
  Ubah orientasi sistem ke MAINNET-FIRST, dengan fallback yang aman.            
                                                                                
  Kerjakan:                                                                     
  1. arsitektur config harus menjadikan mainnet sebagai target production utama 
  2. devnet/testnet tetap boleh ada, tetapi jelas sebagai fallback/development  
mode                                                                            
  3. semua config network harus explicit dan konsisten                          
  4. dashboard/network panel harus menunjukkan:                                 
     - current cluster                                                          
     - whether this is mainnet or non-mainnet                                   
     - operator funding status                                                  
     - tx explorer links sesuai cluster                                         
  5. jika ada perilaku yang hanya aman di devnet, tandai jelas                  
  6. jangan menipu dengan menyebut mainnet jika yang dipakai masih devnet       
default                                                                         
  7. kalau ada code yang auto-airdrop, itu harus dibatasi hanya untuk           
devnet/testnet                                                                  
                                                                                
  Yang penting:                                                                 
  - desain dan wording harus mengarah ke mainnet reality                        
  - jangan jadikan devnet lagi sebagai asumsi default diam-diam                 
                                                                                
  ================================                                              
  PRIORITAS 5 — JUJURKAN PROOF MODEL                                            
  ================================                                              
                                                                                
  Rapikan istilah proof agar tidak misleading.                                  
                                                                                
  Kerjakan:                                                                     
  1. audit semua tempat yang masih menyebut proof seolah itu zero-knowledge     
proof                                                                           
  2. jika implementasi saat ini adalah Ed25519 authorization proof, sebut itu   
dengan jujur:                                                                   
     - operator authorization proof                                             
     - wallet authorization proof                                               
     - cryptographic authorization proof                                        
  3. bedakan dengan jelas:                                                      
     - real implemented cryptographic proof                                     
     - future zk-SNARK / zk circuit proof                                       
  4. update:                                                                    
     - dashboard wording                                                        
     - API metadata                                                             
     - comments                                                                 
     - README/docs bila perlu                                                   
  5. jangan pakai istilah “ZK verified” jika sebenarnya yang diverifikasi adalah
signature authorization                                                         
  6. kalau ada jalur future circuit, labeli sebagai:                            
     - future zk circuit                                                        
     - snark pending                                                            
     - not active yet
